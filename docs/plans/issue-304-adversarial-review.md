# Issue 304 Adversarial Review

## Plan Review

- Dynamic metadata cannot be preserved by ffmpeg static color flags alone.
  The plan requires raw HEVC sidecars and tool-driven injection, which matches
  the external tool workflows.
- The plan rejects unsupported targets before encoding. That avoids expensive
  transcodes that would knowingly drop metadata.
- The plan does not claim synthetic Dolby Vision or HDR10+ fixtures can be
  generated by ffmpeg. Real round-trip validation remains fixture-gated and is
  documented as such.
- The plan keeps the existing DSL surface. `preserve_hdr`, `hdr_mode`, and
  `dolby_vision: copy_rpu` are enough for the requested behavior.

## Code Review

- Silent degradation: dynamic HDR sources with preservation enabled now build a
  `DynamicHdrJob`; missing tools, unsupported Dolby Vision profiles, unsupported
  codecs, and unsupported containers return errors before the transcode starts.
- Cleanup: sidecars live under a single `voom-dynamic-hdr-*` work directory and
  are removed on success and failure. If reinjection fails after the muxed
  ffmpeg temp output is created, that temp output is removed before returning.
- Hardware/software coverage: reinjection happens after the normal ffmpeg
  transcode output is created, so the same sidecar path applies to `libx265`
  and hardware HEVC encoders.
- Remux risk: the remux command copies the injected video plus optional audio,
  subtitle, data, attachment streams, and container metadata from the ffmpeg
  output. Multi-video files are not explicitly modeled beyond replacing the
  first output video stream, which matches VOOM's current single primary video
  transcode assumptions.
- Fixture risk: unit tests verify orchestration and cleanup with fake runners.
  Real HDR10+/Dolby Vision bitstream validation still depends on external
  fixtures and installed tools, as documented in `docs/hdr-transcoding.md`.
